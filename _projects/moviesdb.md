## What this project does

MoviesDB is an end-to-end data pipeline that takes the publicly available IMDb dataset — millions of rows of raw TSV files — and transforms it into a structured, queryable resource. The pipeline ingests, cleans, models, and loads the data into a relational database, then extracts a curated subset of Marvel Cinematic Universe films and exposes them through an interactive graph interface.

The goal was to build something that mirrors a real-world data engineering workflow: raw data in, clean analytical layer out, with automation, testing, and reproducibility at every step.

## Relational Database Design

The foundation of the project is a normalized MySQL schema that mirrors the IMDb data model. Five core tables capture titles, people, crew relationships, ratings, and episode metadata, plus a reference table of Marvel films used for filtering.

### ➕ Marvel Reference Layer

A new **`marvel_movies` table** is generated by scraping the official Marvel Cinematic Universe film list from Wikipedia. This table contains a clean, curated list of known Marvel titles.

This adds a **semantic layer** on top of raw IMDb data.

### What this changes

-  Removes fragile **regex-based filtering**
-  Enables **deterministic joins** (`titles.primary_title = marvel_movies.title`)
-  Makes queries **simpler, faster, and easier to maintain**
-  Improves **reproducibility** (no hidden pattern-matching logic)
-  Allows easy updates when new Marvel movies are released


## SQL Analytics Layer

The most technically involved part of my work was writing the SQL extraction logic. The challenge is that IMDb data doesn't label films by franchise — there's no "Marvel" column. Identifying MCU content requires pattern matching across titles, genres, and years, while filtering out false positives.

Example: extract Marvel movies using regex and genre filtering

```sql
SELECT t.primary_title, t.start_year, r.rating, r.votes
FROM titles t
JOIN ratings r ON t.title_id = r.title_id
WHERE t.primary_title REGEXP 'Iron Man|Captain America|Thor|Avengers|...'
  AND t.title_type IN ('movie', 'tvSeries')
  AND t.genres REGEXP 'Action|Adventure|Sci-Fi'
  AND t.genres NOT REGEXP 'Animation|Crime|Biography'
  AND t.start_year BETWEEN 2008 AND 2026
ORDER BY t.start_year;
```
This approach has now been replaced by a join strategy, using the table scraped from Wikipedia:

```sql
SELECT t.primary_title, t.start_year, r.rating, r.votes
FROM titles t
JOIN marvel_movies m ON t.primary_title = m.title
JOIN ratings r ON t.title_id = r.title_id
ORDER BY t.start_year;
```


The extraction produces seven normalized CSVs — movies, actors, characters, and four relationship tables (plays, appears_in, directs, produces) — ready for direct import into Neo4j.

## Automated Pipeline

The entire workflow is orchestrated by a single Python script (`run_pipeline.py`) that:

1. Downloads and decompresses IMDb data dumps from the official source
2. Spins up a MySQL Docker container and loads the raw data
3. Runs the SQL extraction queries and exports CSVs
4. Optionally loads the results into Neo4j (local Docker or cloud AuraDB)

The pipeline is idempotent — it can be re-run safely whenever IMDb releases updated data. CLI flags (`--createdb`, `--createcsvs`, `--runqueries`) allow running individual stages during development.

## Neo4j Graph Layer

This component was built by [Tarik](https://fr.linkedin.com/in/anouartarik). The graph database models the Marvel universe as nodes (Person, Character, Movie) and relationships (PLAY, APPEAR_IN, DIRECTED, PRODUCED). A Streamlit application provides an interactive front end where users can explore the graph — filtering by actor, movie, or relationship type — with Pyvis rendering the network visualization in the browser.
